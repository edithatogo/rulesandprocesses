# Spec: Engine contributions (PolicyEngine / Axiom / OpenFisca)

## Purpose

Three targeted upstream contributions that solve the host projects' existing pain and, as a side effect, give the PIC contracts rules-heavy-domain consumers. Etiquette rules from `conductor/product-guidelines.md` apply strictly: solve their problem, their idiom, format as footnote. Dylan's insider standing at PolicyEngine/Axiom is the enabler; it is not a license to skip the etiquette.

## C-A: PolicyEngine trace export (issue → PR)

- **Their pain:** PolicyEngine has an internal computation tree used for debugging/explainability, with no stable public contract. Downstream explainers and validators re-derive it ad hoc.
- **Deliverable 1 (investigation):** `external/policyengine/TRACE_INVESTIGATION.md` — where the computation tree lives in `policyengine-core` (cite code permalinks), what it contains per node (variable, period, entity, value, dependencies, parameters touched), and how a scalar/household simulation's tree projects onto the pic-traces step model. **This document is also the project's answer to the vectorized-trace problem**: state precisely what per-case trace derivation is possible (household-level simulations: direct; microsimulation: per-case re-execution or tree projection — measure feasibility, don't assert).
- **Deliverable 2 (issue draft):** propose a versioned `to_trace()` export on household simulations. Value framing: stable contract for explainer UIs, debugging, and external validation (TAXSIM comparisons become path-comparable, not just output-comparable).
- **Deliverable 3 (PR draft, optional if issue lands well):** minimal implementation + tests in policyengine-core idiom.

## C-B: Missingness semantics issue (PolicyEngine and OpenFisca)

- **Their pain:** vectorized engines typically default missing inputs to zero/default, silently conflating "not provided" with "zero". This produces wrong benefit amounts in screener-style usage.
- **Deliverable:** `external/policyengine/MISSINGNESS_CASES.md` + `external/openfisca/MISSINGNESS_CASES.md`: 3–5 concrete failing scenarios each (e.g. SNAP: household with unreported income treated as zero-income → maximum allotment computed, where correct behavior is "cannot determine"), with runnable snippets. Draft issue text asks: is there an idiomatic way to distinguish these today; if not, would a `valueState`-like input annotation be considered? Reference pic-semantics as prior art in a footnote.
- This is deliberately an **issue, not a PR** — the fix touches engine cores and must be maintainer-led.

## C-C: Axiom validation harness

- **Their pain (the real, fundable one):** Axiom generates policy models rapidly with AI; every generated model needs verification against reference implementations. This is the "verification is scarce" thesis applied to Axiom's own pipeline.
- **Deliverable 1:** `external/axiom/HARNESS_DESIGN.md` — a differential validation harness design: Axiom-generated model vs `policyengine-us` (and vs TAXSIM via `policyengine-taxsim` where coverage overlaps) over PIC fixtures; outputs a divergence report (reuse Track 5's report tooling); oracle-independence rule stated (fixtures never generated by the model that generated the code under test).
- **Deliverable 2:** prototype runner in `harness/axiom/` here (against whatever Axiom surface Dylan can expose — this task needs `[HUMAN]` input on Axiom's actual interfaces, which are not public knowledge; the plan gates on it).
- **Positioning:** Axiom is the first consumer, openly. This resolves the capture-risk problem honestly (views/06 §4.3): the harness is built *for* Axiom's QA, published openly, usable by anyone.

## Acceptance criteria

1. TRACE_INVESTIGATION.md cites real code permalinks and gives a measured (not asserted) answer on per-case trace derivation for both household and microsimulation paths.
2. Missingness case files contain runnable reproductions (verified against installed engine versions; record versions).
3. All issue/PR drafts complete in `external/*/SUBMISSION.md` form; submissions themselves are `[HUMAN]`.
4. Axiom harness design reviewed by Dylan (`[HUMAN]`) before any prototype coding.
